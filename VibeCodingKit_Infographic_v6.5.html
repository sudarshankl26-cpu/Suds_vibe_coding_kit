How to use the kit

The soul of it. An AI coding agent has no memory between sessions, so left alone it drifts — it reinvents your types, forgets yesterday's decisions, quietly redefines your words, and says "done" without ever running the code. You can't fix that by asking nicely; the forgetting is built in. So the kit writes the ground truth to plain files on disk that the agent re-reads every session, and — wherever a check has an objective answer — lets one command fail the build. You never "run" the kit; your agent reads it. Your job: lay the files down once, then talk to your agent normally.

It's two phases — and the only thing beginners trip on is the seam between them. First a tiny one-time setup you do by double-clicking (no AI, no terminal wizardry, ~5 min). Then you open your AI agent and just work — but your first message isn't "build me a …", it's the scope-setting grill (② below). Get that one message right and the rest follows.

① One-time setup

Before you open any AI tool · ~5 min · just you

Don't overthink these — they only put the files in place. Nothing here can harm your computer, and you don't need to understand any of it yet.

1
Install mise — once ever, on your computer. winget install jdx.mise (Windows) or curl https://mise.run | sh (WSL). It quietly fetches the tools the gates need so you never have to. On later projects it's already there — skip it.
2
Make a folder, name it your project, extract the kit into it. Files go straight in — no sub-folder. This folder IS your app.
3
Double-click scaffold.ps1. (WSL: bash scaffold.sh.) Sets up the toolchain, git, and the safety gates, makes the first save-point, and prints your first prompt. Safe to run twice.
That's the entire setup — and you still haven't needed the AI, or read a single rule file.

② Open your agent — the first message

The one step people skip

Open your AI coding agent in the folder. Don't start with "build me a …". Your first message defines the scope and runs the grill-me interview: it asks one sharp question at a time and turns your answers into a spec (PRD.md), so the agent builds the right thing instead of guessing.

Paste this as your very first message Read CONTEXT.md and the files it lists. Then run the grill-to-prd skill
(skills/grill-to-prd.md): interview me ONE question at a time, recommending
an answer for each, until the scope is fully clear. # do NOT write code or scaffold yet
Then write PRD.md and we'll build phase by phase.
Why it matters: skip the grill and the agent invents the requirements — the #1 way it builds the wrong thing. Five minutes of questions here saves hours of rework.

③ The daily loop — what you'll actually do

After the first session, every session is the same short loop. You don't manage the agent's bookkeeping — you point it at files and let it run, trusting the green gate.

Orient

Read RepoMap + STATE — don't re-walk the tree

Track

Real deliverables → OpenTasks (not process steps)

Build

One phase; just verify freely (stays green)

Verify-all

just verify-all before commit — tasks + smoke

Record

WorkLog + STATE; bug → SinsGotchas prose (rule if structural)

When you start a session: paste Re-orient (below). The agent reads the map + cursor and picks up exactly where it left off — no re-walking, no "let me check every file."
While you build: run just verify any time. It's the fast gate (lint/types/tests/libs/schema/sins/dox) and stays green mid-work — use it to sanity-check, not to gate yourself.
Before you commit: just verify-all. It adds the bookkeeping + a Playwright smoke. Green here means "ready," not "probably fine."

Commit cadence

Two-level discipline, no ceremony

You don't commit every keystroke. Commit when something is objectively true:

1
Per phase. A phase ends with just verify-all green + WorkLog/STATE updated. That's a commit.
2
When contracts change. PRD / GLOSSARY / DATA_MODEL / ARCHITECTURE edits commit separately from code (the agent does this for you under §H).
3
Before risky work. A refactor, a library swap, an irreversible migration — tag first: git tag pre-<thing>.
The agent handles its own bookkeeping. WorkLog, STATE, SinsGotchas entries — you don't write them. You just review the diff before merging.

Why it works — the theory

It externalizes the memory the agent doesn't have. Files named so the filename itself signals the purpose — OpenTasksMustCompleteAll, RepoMapReadFirst, SinsGotchasLearnings. The agent re-reads them every session; nothing lives only in chat.
It converts the few objectively-checkable rules into gates. A rule in a doc is a suggestion; the same rule as a just verify check is a wall. The gate describes objective truth about the product, never the agent's bookkeeping.
It gates the product, not the process. The kit refuses to spend tokens proving the agent followed a workflow. v6.1 explicitly removed that failure mode; every later version has held the line.

Names are part of the prompt. Blunt, purpose-revealing filenames steer the model better than bland ones — leave them verbatim in your prompts.
Don't gate your own prose. If you're editing a log or a timestamp to make a gate pass, stop — that's the v6 failure mode v6.1 removed.
Trust the green gate. If just verify passes, the known patterns, APIs, contracts, and types are clean. Re-auditing by hand just burns tokens.

What's new in v6.5

Three real-world failure modes, closed — and all still true to "gate the product, not the process": no new per-turn ceremony.

No silent false-greensA gate whose tool is missing now FAILS (exit 3) instead of printing "skipping" and passing. The JS package manager is detected from the lockfile (never assumed). New just doctor self-tests the harness — a green doctor is what makes a green verify trustworthy.
Docs that can't rot silentlydox-check now fails on wholesale-stale path references in contract docs and on leftover [fill in] placeholders in RepoMapReadFirst — an agent that catches one stale doc rationally stops trusting all of them.
The learning loop gets a motorSins entries carry retrieval metadata (Scope/Trigger/Enforced/Digest/Recurrences); the index has a ★ALWAYS block + a trigger map + enforced entries collapsed out of the read path. The recurrence ratchet: a repeat is never re-documented — promote it (type → rule → test → digest).
Subagent economics (§4b)A context firewall (subagents write to .scratch/<task>/, return path + 3-line summary — never raw dumps), handoff packets, and a stakes/reversibility/ambiguity floor for routing slices across model tiers.
CRLF-immune gatesEvery gate normalizes \r\n → \n on read, so a Windows editor that saves CRLF can no longer silently break the regexes that anchor on \n.
Migration tooling documentedv6.4→v6.5 has a surgical manual swap (a dozen infra files); older projects use the classify/backup migrator that never overwrites your state.
